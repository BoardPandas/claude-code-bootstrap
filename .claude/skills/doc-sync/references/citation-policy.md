# Evidence and Citation Policy

Every substantive claim in a generated page must be backed by a citation pointing to a real line range in a real file. This is the only thing that distinguishes generated docs from confident hallucination.

## Hard rules

1. **Never guess line numbers.** Use the line numbers Read returns. If you cannot locate the line, do not cite the file.
2. **Never invent file paths, function names, or behavior.** If a claim has no source, write `_TBD_` and a one-line reason instead.
3. **Only cite repo files.** Do not cite documentation generated by this skill (no self-citations).
4. **Do not cite trivia.** Skip citations for general programming concepts, standard library calls, or your own framing sentences.

## URL format

Use repo-relative paths in markdown links so the docs remain valid in the working tree, on GitHub, and on any web renderer that resolves relative paths.

```
[file.ext:42](path/to/file.ext#L42)
[file.ext:42-50](path/to/file.ext#L42-L50)
```

If `Docs/_toc.yaml` defines `repo_base_url` and `ref_commit_hash`, you may emit absolute URLs instead. Pick one style per page and stick with it.

```
[file.ext:42-50](https://github.com/owner/repo/blob/{commit}/path/to/file.ext#L42-L50)
```

### Display text

| Component | Rule | Example |
|---|---|---|
| Filename | Use just the basename, not the full path | `Button.tsx` not `src/components/Button.tsx` |
| Line numbers | Real numbers from the file | `:42` or `:42-50` |
| Separator | Colon between filename and lines | `Button.tsx:42` |

## Inline citations

Place a citation immediately after the claim it supports, wrapped in parentheses, and BEFORE the closing period of the sentence:

- Right: `The middleware rejects unauthenticated requests with 401 ([auth.ts:84-92](src/middleware/auth.ts#L84-L92)).`
- Wrong: `The middleware rejects unauthenticated requests with 401. [auth.ts:84-92](src/middleware/auth.ts#L84-L92)`
- Wrong: `The middleware rejects unauthenticated requests with 401 [auth.ts:84-92](src/middleware/auth.ts#L84-L92).`

The parentheses make the citation visually separable from the prose.

## Citations in tables

Wrap citations in parentheses inside table cells too:

```markdown
| Endpoint | Method | Description |
|---|---|---|
| `/v1/auth/login` | POST | Issue a session token ([login.ts:1-40](src/routes/login.ts#L1-L40)) |
| `/v1/auth/logout` | POST | Revoke the current session ([logout.ts:1-22](src/routes/logout.ts#L1-L22)) |
```

## End-of-section citations

End each section with a `Sources:` line summarizing every file cited in that section. This gives readers a one-glance reference list per topic and is the primary citation mechanism when a section is mostly tabular.

```markdown
Sources: [auth.ts:1-120](src/middleware/auth.ts#L1-L120), [session.ts:40-90](src/lib/session.ts#L40-L90)
<!-- END:AUTOGEN myapp_02_auth_middleware -->
```

If every paragraph already has inline citations and the `Sources:` line would be a verbatim repeat of three or fewer files, you may omit it.

## Distribution

- Spread citations across the section. Do not stack five citations at the bottom of a paragraph.
- Each major claim or table row should have a citation nearby.
- A code block lifted from a source file must carry a citation either above or below it.

## Code snippets

When you embed a code excerpt:
- Lift it verbatim (excluding any line-number prefix Read added)
- Use a fenced block with the language identifier
- Cite the line range immediately above OR immediately below the block
- Prefer real excerpts over hypothetical examples; only synthesize examples to illustrate documented public APIs

```typescript
// src/middleware/auth.ts:84-92
export const requireSession = async (req, res, next) => {
  const session = await getSession(req);
  if (!session) return res.status(401).end();
  req.session = session;
  next();
};
```

Sources: [auth.ts:84-92](src/middleware/auth.ts#L84-L92)

## What NOT to cite

- General programming concepts ("HTTP is stateless")
- Standard library functions when not modified
- Your own summaries and framing sentences
- The Mermaid diagram itself (citations go in surrounding text, not inside the diagram)

## Missing evidence

If you cannot find a source for a claim that the page template requires (e.g., the template demands an "Error Handling" section but the code does no explicit error handling), write:

```markdown
## Error Handling

_TBD_ — no explicit error-handling code found in [src/foo/](src/foo/) at commit {hash}. Add a section here once handlers are introduced.
```

This keeps the section structurally complete and makes the gap searchable.
